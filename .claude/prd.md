# Content-OS 需求书 v1.0

> SaaS 从业者专用的自媒体六边形系统 — 用架构师思维把自媒体变成可部署、可迭代、可扩展的 SaaS 系统

## 0. 用户画像

| 角色 | 核心场景 | 痛点 | 与系统的触点 |
|------|---------|------|------|
| SaaS 创始人 | Build in Public、产品发布、技术写作 | 没有时间做内容，发布不系统 | 选题库→AI草稿→一键发布→数据复盘 |
| 独立开发者 | 技术博客、教程、开源推广 | 写作效率低，分发渠道管理混乱 | AI 辅助写作→多平台分发→转化追踪 |
| 自媒体运营 | 矩阵账号管理、内容排期、数据复盘 | 多平台切换频繁，数据散落各处 | 内容中台→排期发布→多平台数据聚合 |

## 0.5 核心使用场景

### 场景1: 从选题到发布的快乐路径 (SaaS 创始人)

1. 打开选题库，查看当日推荐话题 → 对应 R1 选题库 (A1)
2. 选中话题"我如何用六边形架构重构 SaaS"，一键生成大纲 → 对应 R2 AI 文案 (B1)
3. AI 生成初稿，人工微调 → 对应 R3 内容流水线 (A2)
4. 选择发布平台（公众号+掘金+知乎），定时发布 → 对应 R6 分发 (C1)
5. 发布完成，48h 后查看数据报告 → 对应 R9 数据聚合 (D1)

### 场景2: 发布失败的异常路径

1. 定时发布任务触发 → 对应 R6 (C3)
2. 平台 API 返回 `rate_limit` → 系统自动重试 3 次 (指数退避) → 对应 converge-core CircuitBreaker
3. 重试仍失败 → 写入死信队列，通知用户手动处理 → 对应 converge-core RetryHandler
4. 用户查看失败原因，修正后手动发布 → 回到场景1

## 1. 问题陈述

**当前状态**: SaaS 创始人做自媒体靠灵感和手工操作：记事本写选题→Word/Notion 写稿→手动复制粘贴到各平台→切换不同后台看数据。

**核心问题**: 自媒体运营缺乏系统化工具——选题靠灵感赌运气、多平台分发纯手工、数据散落无法归因。现有工具（如稿定、微小宝）定位通用自媒体，不解决 SaaS 垂直领域的专业需求。

**影响范围**: SaaS 创始人和开发者群体，每人每天浪费 1-2h 在重复性内容操作上。

## 2. 方案对比

| 维度 | 方案A: 用现成工具组合 | 方案B: 自建垂直SaaS | 推荐 |
|------|------|------|:--:|
| 启动成本 | 低（按月订阅） | 中（2 个月开发 MVP） | — |
| 垂直适配 | 差（SaaS 选题库、话术模板需手动维护） | 强（内置 SaaS 领域知识） | ✅ B |
| 差异化 | 无（所有人用同样工具） | 强（唯一"架构师做自媒体"定位） | ✅ B |
| PLG 飞轮 | 无（工具和内容脱节） | 有（内容养工具，工具反哺内容） | ✅ B |
| 长期成本 | 递增（按席位付费） | 递减（自建边际成本趋零） | ✅ B |
| 数据所有权 | 归平台 | 归自己 | ✅ B |

**推荐**: 方案 B — 自建垂直 SaaS。PLG 飞轮是决定性优势：每篇内容都是产品 demo，每个用户都是内容传播节点。

## 3. 架构设计

### 六边形架构

```
┌── Input Ports ──────┐  ┌── Domain Core ──────┐  ┌── Output Ports ────┐
│ 选题端口 (A1)        │  │ 个人IP: 架构师做SaaS  │  │ 内容适配器 (C1-C4)  │
│ 素材端口 (A2)        │  │ 内核: 系统思维>流量   │  │ 平台适配器 (C1-C4)  │
│ 灵感端口 (A3)        │  │ 价值: 自媒体=可复用系统│  │ 变现适配器 (E1-E4)  │
└─────────────────────┘  └──────────────────────┘  └────────────────────┘
         │                        │                         │
         └──────────┬─────────────┴──────────┬──────────────┘
                    │  Infrastructure (Foundation)  │
                    │  AI流水线 · 数据闭环 · 矩阵管理  │
                    │  converge-core: Hooks/Auth/CB │
                    └─────────────────────────────┘
```

### 模块清单

| 模块 | 职责 | 子模块 |
|------|------|------|
| A: 内容中台 | 选题+素材+流水线+合规 | A1选题库 A2素材库 A3流水线 A4合规 |
| B: AI 适配器 | 标题/文案/封面/回复 | B1标题 B2文案 B3封面 B4回复 |
| C: 分发适配器 | 多平台发布+排程 | C1授权 C2排程 C3适配 C4重试 |
| D: 数据适配器 | 聚合+漏斗+归因+报告 | D1聚合 D2漏斗 D3归因 D4报告 |
| E: 变现适配器 | 会员+付费+分销+私域 | E1会员 E2课程 E3分销 E4私域 |
| F: 权限安全 | 角色+加密+审计+合规 | F1权限 F2加密 F3审计 F4备案 |

### 关键设计决策

1. Domain 实体零 IO — 所有业务规则在纯 PHP 对象中，不依赖框架
2. 跨模块通信走 Hooks — 模块间不直接 `use` 对方类
3. AI 能力通过 Port-Adapter 接入 — 不绑定具体 LLM Provider
4. converge-core 提供全部基建 — 认证/熔断/日志/断言/门禁
5. 前端 Vanilla JS + Alpine.js — 复用 converge-ui 设计令牌

| Phase | 时间 | 交付 | 验收 |

| P5: 变现 | 第 9-10 周 | E1-E4 + F1-F4 | 会员订阅+权限完整可用 |
| GA | 第 12 周 | 全部模块 + 内测 50-200 人 | PLG 飞轮运转 |
